# Личная CRM для дизайн-студии

Полнофункциональная CRM для ведения клиентов, заказов, финансов и аналитики
дизайн-студии. Рассчитана на одного владельца, но архитектура готова к
добавлению сотрудников.

## Стек технологий

- **Next.js 16** (App Router) + **TypeScript**
- **Tailwind CSS v4** + компоненты в стиле shadcn/ui (написаны вручную)
- **Drizzle ORM** + **PostgreSQL** (Supabase) — работает одинаково локально и в проде
- **NextAuth v5** — авторизация единственного владельца CRM
- **Recharts** — графики на дашборде
- **React Hook Form + Zod** — формы и валидация
- **date-fns** — работа с датами (формат ДД.ММ.ГГГГ)
- **Vitest** — тесты финансового модуля

> **Почему Drizzle, а не Prisma?** Технически задание допускало «Prisma или
> другой удобный ORM». Prisma требует скачивания бинарных движков при `generate`/`migrate`,
> и в среде, где собирался этот проект, доступ к серверам Prisma был закрыт.
> Drizzle работает как обычная TypeScript-библиотека без внешних бинарников,
> поэтому миграции гарантированно применяются в любом окружении, включая
> оффлайн/закрытые корпоративные сети.

## Быстрый старт

```bash
# 1. Установить зависимости
npm install

# 2. Настроить переменные окружения
cp .env.example .env
# откройте .env и задайте DATABASE_URL/DIRECT_URL (строки подключения
# Supabase — Project Settings → Database → Connection string, см. ниже),
# а также свой OWNER_EMAIL / OWNER_PASSWORD / AUTH_SECRET

# 3. Создать базу данных и применить миграции
npm run db:migrate

# 4. Заполнить демонстрационными данными (необязательно, но рекомендуется)
npm run db:seed

# 5. Запустить в режиме разработки
npm run dev
```

Откройте [http://localhost:3000](http://localhost:3000) — вы попадёте на
страницу входа. Используйте email/пароль, указанные в `.env`
(по умолчанию `owner@studio.local` / `change-me-please` — **обязательно
смените их перед реальным использованием**).

### Сборка для продакшена

```bash
npm run build
npm run start
```

### Тесты

```bash
npm test
```

Покрывает финансовый модуль (`lib/calculations`): прибыль, остаток к оплате,
маржинальность, изменение показателей между периодами, просрочку,
агрегацию и сравнение месяцев — 28 тестов.

### Полезные npm-скрипты

| Команда | Назначение |
|---|---|
| `npm run dev` | запуск в режиме разработки |
| `npm run build` | production-сборка |
| `npm run db:generate` | сгенерировать новую миграцию после изменения `lib/db/schema.ts` |
| `npm run db:migrate` | применить миграции к базе данных |
| `npm run db:seed` | заполнить демонстрационными данными (очищает текущие данные!) |
| `npm run db:reset` | пересоздать базу с нуля и засеять заново |
| `npm test` | прогнать тесты |

```
  calculations/       чистый модуль финансовых расчётов + тесты

```

## Функциональность

- **Клиенты**: карточки с контактами, источником, тегами, статусом; таймлайн
  взаимодействия (звонки, встречи, предоплаты, отзывы и т.д.); история
  заказов и агрегаты (выручка, прибыль, задолженность).
- **Заказы**: таблица (сортировка, группировка по месяцу/статусу/клиенту,
  массовые действия, дублирование, экспорт выбранного), Канбан с
  drag-and-drop, Календарь по дедлайнам/датам.
- **Дашборд**: KPI с сравнением периодов, 5 графиков, воронка проектов,
  «Лучшие месяцы», «Тенденции» — всё считается из реальных данных, без
  заглушек.
- **Аналитика**: фильтры по услуге/источнику/статусу, сравнение месяцев с
  переходом к списку заказов периода.
- **Настройки**: редактируемые справочники (услуги, источники, статусы
  проекта с цветом и порядком, статусы оплаты), профиль CRM, импорт CSV/XLSX
  с сопоставлением колонок и обработкой дублей, экспорт CSV/XLSX/JSON,
  полное резервное копирование и восстановление, безопасное удаление данных.
- Тёмная/светлая тема, горячая клавиша **N** — новый заказ, **⌘K** —
  глобальный поиск, персистентные фильтры между переходами по страницам.

## Подключение к Supabase (PostgreSQL)

БД — обычный Postgres-проект Supabase. В Project Settings → Database →
Connection string нужно скопировать **два** варианта строки подключения:

- **Transaction pooler** (порт `6543`) → `DATABASE_URL` — используется
  приложением в рантайме (в т.ч. на Vercel). Важно: клиент `postgres-js`
  в `lib/db/index.ts` создаётся с `{ prepare: false }`, т.к. PgBouncer в
  transaction-режиме не поддерживает prepared statements.
- **Session pooler** (порт `5432`, тот же хост) → `DIRECT_URL` —
  используется только для миграций (`drizzle-kit`, `scripts/migrate.ts`),
  которым нужна сессия с сохранением состояния.

Обе строки указывают на одну и ту же базу, просто на разные точки входа
пулера Supabase. Заполните оба значения в `.env` (см. `.env.example`).

## Известные упрощения

Честно о том, что сделано с сокращённым объёмом ради работающего результата:

- **Ширина колонок таблицы заказов** не перетаскивается мышью (есть
  скрытие/показ колонок через фильтры, но не drag-resize).
- **Импорт** заточен под колонки из ТЗ (Заказ, Выручка, Расходы, Заказчик,
  Отзыв, Статус, Откуда пришёл, Дата контакта, Дата исполнения) — для заказов;
  массовый импорт клиентов отдельным мастером не реализован (создаются
  автоматически при импорте заказов по имени).
- **Права доступа сотрудников** — в схеме есть поле `role`, но экран
  управления сотрудниками не реализован (только один владелец), как и
  указано в ТЗ на первом этапе.

Все расчёты (прибыль, остаток, маржинальность, просрочка, сравнение периодов)
покрыты тестами и не содержат моков или случайных чисел.
